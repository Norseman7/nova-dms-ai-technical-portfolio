# Audit And Handover Workflow

## Audit Workflow

DMS audit templates are structured governance tools. They define:

- audit purpose
- target scope
- explicit non-scope
- required read order
- protected areas
- inspection focus
- report sections
- deferred human checks
- archive/report routing
- completion criteria

Audits validate and report. They do not automatically remediate, reorganize, or mutate the system unless a separate scoped task authorizes implementation.

## Audit Categories

| Audit type | What it protects |
| --- | --- |
| Runtime hygiene | runtime correctness and regression detection |
| Governance | authority structure, document placement, lifecycle coherence |
| Report integrity | deterministic reporting and metric correctness |
| UI runtime truth | visible runtime behavior and user-facing correctness |
| Deep audit | systemic integrity, architecture coherence, long-term resilience |
| Repository audit | storage hygiene, lifecycle separation, structural integrity |
| Architecture protection | boundaries, ownership, scalability, and drift risk |

## Handover Workflow

Handovers preserve deterministic continuity between development sessions. A strong handover states:

- context
- current authoritative state
- completed scope
- affected files or components
- locked decisions
- known limitations
- next exact task
- what not to do next
- acceptance criteria
- start prompt for the next session
