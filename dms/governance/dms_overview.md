# DMS Overview

DMS is the development management layer used to support long-running project control. It provides evidence of documentation discipline, audit workflows, handover continuity, project-scoped context, and AI-assisted development governance.

## Purpose

DMS exists to reduce drift across complex work by making project context, documents, audits, handovers, reports, and task state easier to review and control.

## Capabilities Represented

| Capability | Evidence |
| --- | --- |
| Project-scoped control | DMS tracks active project context and project-specific documentation paths. |
| Document access | Documents are organized by categories such as handovers, specs, as-built records, authority, governance, thread initializations, validation, and debug material. |
| Handover continuity | Latest handovers and controlled transition records preserve context between work sessions. |
| Audit discipline | Audit templates define purpose, scope, non-scope, report structure, archive path, and deferred human checks. |
| Task tracking | Manual task tracking records task category, comments, start/end timing, and handover references. |
| Report generation | Deterministic reporting supports review without relying on AI-generated claims. |
| Agent/governance control | Prompt and agent files define operating discipline for ChatGPT and Codex collaboration. |

## Portfolio Relevance

DMS shows that Paul did not rely on ad hoc AI-assisted development. He built a governance layer around the work: task boundaries, audit categories, documentation locations, handover protocols, and review gates.

## Relationship To NOVA

DMS is not the NOVA system itself. It is the control layer that helps NOVA remain coherent while the architecture grows across data acquisition, DATABASE, ANALYSIS, event modules, Brain Store planning, UI, and validation.
